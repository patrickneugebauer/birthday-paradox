# Project Structure

## Top-Level Directories

| Path | Contents |
|---|---|
| `solutions/` | One folder per language — each contains Dockerfiles with names following the pattern `Dockerfile[.{runtime}][.{data_structure}][.{execution_method}]` and source files for a Monte Carlo birthday-paradox simulation |
| `cmd/app/` | Go CLI entry point (`main.go`) — dispatches commands: `map-files`, `stars`, `build`, `weigh`, `run`, `readme` |
| `internal/tasks/` | Go task implementations — one file per command, plus `constants.go` and `io.go` |
| `reference/` | Static input data: `language-info.csv` (language metadata) |
| `artifacts/` | Generated output files (git-ignored): JSONL results, `.sh` command logs, `readme.md` |
| `api-sample-data/` | Sample raw API responses used for development/reference (e.g. `ghub-golang.json`) |
| `plans/` | Implementation plan files (`YYYYMMDD-short-description.md`) |

## Key Files

| File | Description |
|---|---|
| `reference/language-info.csv` | Language → year, wiki URL, GitHub URL |
| `artifacts/dockerfiles.jsonl` | `{language, dockerfile, runtime, data_structure, execution_method, tag}` — generated by `map-files` |
| `artifacts/build-results.jsonl` | `{tag}` — built Docker image tags |
| `artifacts/weigh-results.jsonl` | `{tag, size_mb}` — image sizes |
| `artifacts/run-results.jsonl` | `{image, iterations, sample_size, percent, seconds, ips}` — benchmark results |
| `artifacts/star-results.jsonl` | `{language, stars}` — GitHub star counts |
| `artifacts/star-data.json` | Raw GitHub API repo objects (richer than star-results) |
| `config.json` | Legacy per-language config (JSON array, not used by current CLI) |

## Dockerfile Naming Convention

Dockerfiles follow the pattern: `Dockerfile[.{runtime}][.{data_structure}][.{execution_method}]`

**Components:**
- `runtime`: The language implementation (compiler, interpreter, VM). Examples: `bash`, `clang`, `gcc`, `jvm`, `node`, `dotnet`
- `data_structure`: The algorithm variant used in the solution. Examples: `array`, `list`, `vector`, `set`. When unused, denoted as `-` (omitted from filename)
- `execution_method`: Optional method for executing the same language/runtime/data_structure. Examples: `build`, `publish`, `run`, `mix-build`, `ocamlopt`. Only included when the same combination can be executed multiple ways

**Docker Tags:** The `file-tree.go:getRuntimeAndTag()` function splits the filename on `.` and generates Docker tags in the format `bday/{language}:{runtime}[.{data_structure}][.{execution_method}]` with all tag components lowercased.

Examples:
- `bday/bash:bash.associative-array`
- `bday/c:clang.array`
- `bday/csharp:dotnet.array.build`
- `bday/chapel:chapel` (when data_structure = `-`)
